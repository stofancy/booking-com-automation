# OpenClaw Skill 开发最佳实践

本文档旨在为 AI Coding Agent 提供一套清晰、可操作的最佳实践指南，用于开发安全、高效且易于维护的 OpenClaw Skill。遵循这些原则，你的 Agent 可以创建出高质量的工具，并有效规避社区中已发现的风险。

---

## 1. 核心设计原则

### 1.1 职责单一，指令精准
- **原则**：一个 Skill 应只专注于完成一件具体的事情（如“查询天气”、“创建 GitHub Issue”），而非大而全的模糊功能。
- **实践**：在 `SKILL.md` 中，使用清晰、简洁的指令告诉模型 **“做什么”**，而不是长篇大论地教它“如何成为一个 AI”。指令应直接引导模型调用具体的工具或执行明确的步骤（参考 [官方文档](https://docs.openclaw.ai/tools/creating-skills)）。

### 1.2 最小权限原则
- **原则**：Skill 应仅申请完成其功能所必需的、最低限度的权限。过度申请权限是重大的安全漏洞。
- **实践**：
    - 优先考虑“纯指令型”Skill（即只包含 `SKILL.md` 文件，不包含任何可执行代码），这是风险最低的模型。
    - 明确 Skill 到底需要操作什么。例如，只需要读取文件，就不要申请写入权限。
    - 如果可以通过现有系统工具（如 `bash`、`browser`）完成任务，应优先利用它们。

## 2. 结构与实现规范

### 2.1 遵循标准 Skill 结构
一个能被 OpenClaw 内核正确识别和调度的 Skill，必须遵循以下结构规范：

```
your-skill-name/
├── SKILL.md # (必需) 包含 YAML frontmatter 的指令文件，定义元数据和核心逻辑
├── index.js/ts # (可选) 核心执行逻辑，当需要代码时使用
├── plugin.json # (可选) 当需要代码时，此文件声明元信息、参数和权限
└── package.json # (可选) 声明 Node.js 依赖
```

### 2.2 指令与代码分离，输入输出标准化
- **原则**：保持 `SKILL.md` 中的指令与 `index.js` 中的执行逻辑分离，使 Skill 更容易理解和维护。同时，Skill 向内核返回的结果必须是标准化的。
- **实践**：
    - 在 `SKILL.md` 中，用自然语言描述何时以及如何使用这个 Skill，并给出具体的命令示例（参考 [GitHub Skill 示例](https://clawhub.ai/steipete/github)）。
    - 在 `index.js` 中，实现具体的功能，并确保 `run` 函数返回统一格式的结果对象，通常包含 `success`、`message` 和 `data` 等字段。

### 2.3 无状态设计
- **原则**：Skill 本身不应保存任何用户状态或会话数据。所有必要的上下文信息都应由 OpenClaw 内核在每次调用时作为参数传递。
- **实践**：避免在 Skill 内部使用全局变量存储用户特定信息。所有需要的数据都从 `params` 参数中获取。

## 3. 安全开发规范（核心关注点）

近期社区安全分析显示，ClawHub 上存在恶意 Skill，它们通过各种手段窃取数据或控制用户系统。**安全是 Skill 开发的首要考量。**

### 3.1 严控外部资源下载
- **高风险行为**：绝对避免在 Skill 的安装说明或代码中，要求用户执行 `curl [invalid URL removed]` 或下载运行来历不明的二进制文件、密码保护的压缩包。这是目前最主流的恶意攻击手法。
- **最佳实践**：
    - Skill 的所有功能应尽可能通过内置工具或明确、可信的依赖实现。
    - 如果需要依赖外部工具，应优先使用系统包管理器（如 `apt`、`brew`）安装官方版本，或在文档中清晰说明用户需自行安装（参考 [GitHub Skill 的依赖说明](https://clawhub.ai/steipete/github)）。

### 3.2 警惕命令注入
- **原则**：如果 Skill 需要构造 shell 命令，必须对用户的输入进行严格的验证或转义，防止恶意用户通过输入特殊字符来执行未授权的命令。
- **实践**：尽量使用内置的 API 或库函数来完成操作，而不是拼接字符串执行 shell 命令。如果必须使用 `bash` 等工具，要使用参数化的方式或对输入进行严格过滤。

### 3.3 谨慎处理敏感数据
- **原则**：确保 Skill 执行过程中接触到的 API 密钥、文件内容等敏感信息不会被意外泄露或持久化到日志文件中。
- **实践**：
    - 对于必需的外部服务 API 密钥，指导用户通过环境变量或 OpenClaw 的安全配置机制注入，而不是硬编码在 Skill 文件中。
    - 避免将敏感的命令输出（如 `CLAUDE_TOOL_OUTPUT`）不加处理地写入持久化文件（参考 [self-improving-agent 的安全建议](https://clawhub.ai/pskoett/self-improving-agent)）。

### 3.4 保护持久化上下文
- **原则**：像 `AGENTS.md`、`SOUL.md` 这类文件定义了 Agent 的核心行为和规则，恶意 Skill 可能会篡改这些文件植入后门指令。
- **实践**：如果你的 Skill 需要修改这类文件，必须在文档中明确告知用户其行为，并尽可能获得用户的明确确认。普通用户则应像保护 SSH 密钥一样保护这些文件。

## 4. 测试与发布

### 4.1 全面的本地测试
- **实践**：在将 Skill 发布到社区前，务必使用命令 `openclaw agent --message "use my new skill"` 来模拟实际调用，测试各种正常和异常的输入情况，确保功能正常且无意外行为（参考 [官方文档](https://docs.openclaw.ai/tools/creating-skills)）。

### 4.2 提供清晰的使用文档
一个易于使用的 Skill 离不开清晰的文档。在 `SKILL.md` 或项目的 `README.md` 中，至少应包含以下信息：
- **清晰的名称和描述**：让用户一眼就能明白 Skill 的用途。
- **前置条件 (Prerequisites)**：明确列出所需的外部依赖（如 `gh` CLI、某个 API Key）及其配置方法。
- **使用示例**：提供多个常用的命令示例，用户可以直接复制使用（参考 [GitHub Skill 示例](https://clawhub.ai/steipete/github)）。
- **权限说明**：明确声明 Skill 会用到哪些系统工具和权限。

### 4.3 进行安全验真
- **实践**：作为 Skill 的发布者，有责任确保自己创造的工具不会危害用户。在发布到 ClawHub 前，可以自行或利用社区工具（如 **VirusTotal** 的 **Code Insight** 功能）对 Skill 包进行扫描，重点检查是否存在上述提到的恶意或高风险行为（参考 ClawHub 页面的安全扫描部分）。
